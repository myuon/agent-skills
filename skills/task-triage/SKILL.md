# Task Triage: タスクの振り分けスキル

タスク一覧を分析し、実装可能性に基づいて振り分けを行う。

## 手順

### 1. タスク一覧を取得

```bash
gh issue list
```

各 Issue の詳細を確認:

```bash
gh issue view <number>
```

### 2. 各タスクを判定

以下の基準で「実装可能」か「リファイン必要」かを判定する。

#### 実装可能の条件（すべて満たす）

- [ ] 何を作るか・変更するかが具体的に記述されている
- [ ] 受け入れ条件または完了条件が明確である
- [ ] 技術的な実現方法に大きな不確実性がない
- [ ] 影響範囲が特定できる

#### リファイン必要の条件（いずれかに該当）

- 要件が曖昧・抽象的で複数の解釈が可能
- ユーザーの判断・選択が必要な設計上の分岐がある
- 前提条件や依存関係が不明
- スコープが大きすぎて分割が必要

### 3. 振り分け結果をユーザーに提示

以下の形式で一覧を提示し、ユーザーの確認を得る:

```
## 振り分け結果

### 実装可能
- #<number>: <title> — <実装可能と判断した理由>

### リファイン必要
- #<number>: <title> — <リファインが必要な理由・確認したいポイント>
```

**ユーザーの確認を得てから次のステップに進むこと。**

### 4. 実装可能なタスクを並列実装

ユーザーの承認後、実装可能なタスクを Agent ツールで並列に実装する。

各 Agent には以下を指示:

- 対象 Issue の番号と内容
- ブランチ名（`issue-<number>-<slug>` 形式）
- 実装後に commit し、PR を作成すること
- PR の本文に `Closes #<number>` を含めること

```
Agent: Issue #<number> の実装
- main から新しいブランチを作成
- 実装を行う
- テストがあれば実行
- commit & push
- PR を作成（Closes #<number>）
```

### 5. リファインが必要なタスクを対話的に詳細化

リファインキューのタスクについて、ユーザーと対話しながら仕様を固める:

1. 不明点・選択肢を提示する
2. ユーザーの回答をもとに仕様を具体化する
3. 仕様が固まったら Issue を更新する

```bash
gh issue edit <number> --body "$(cat <<'EOF'
## 更新された仕様
...
EOF
)"
```

4. 仕様が確定したタスクは実装キューに移し、ステップ 4 と同様に実装を開始する
